# OBProxy load test, 2026-05-25

## Scope

Client:

- exp2, using the system `mysqlslap` and `mysql` clients.

Target:

- OceanBase CE 4.5.0.0 cluster `alipay-ob`.
- OBProxy `10.190.49.117:2883` and `10.190.14.135:2883`.
- Tenant/database `alipay_tenant/alipay_demo`.

Workload:

- Dedicated tables `loadtest_accounts` and `loadtest_ledger`.
- `point_read`: random primary-key balance lookup.
- `transfer`: stored procedure transaction with two row locks, two balance updates, and one ledger insert.

Script:

```bash
scripts/loadtest-obproxy.sh
```

## Dual OBProxy burst

Command shape:

```bash
MODE=prepare HOST=10.190.49.117 scripts/loadtest-obproxy.sh
HOST=10.190.49.117 MODE=write CONCURRENCY_LIST=32 QUERIES=50000 scripts/loadtest-obproxy.sh &
HOST=10.190.14.135 MODE=write CONCURRENCY_LIST=32 QUERIES=50000 scripts/loadtest-obproxy.sh &
wait
```

Results:

| OBProxy | Clients | Queries | Avg sec | TPS |
|---|---:|---:|---:|---:|
| 10.190.49.117:2883 | 32 | 49984 | 17.264 | 2895.27 |
| 10.190.14.135:2883 | 32 | 49984 | 17.264 | 2895.27 |
| combined | 64 | 99968 | 17.264 | 5790.54 |

Integrity check:

| Check | Value |
|---|---:|
| ledger_rows | 99968 |
| total_balance | 100000000.00 |

Post-test state:

- All three OBServer nodes remained `ACTIVE`.
- Both OBProxy nodes remained `active`.
- No deadlock, timeout, or client error was reported by the script.

## Reading

This is a lab baseline, not a capacity certificate. The client ran from exp2, which is also an OBServer node, and the workload uses `mysqlslap` plus a stored procedure. The current result is still useful: the OBProxy path is stable, consistency holds under concurrent writes, and this small transactional workload plateaus around 5.6k to 5.8k TPS/QPS from this client setup.
